## 一个奇怪的场景

我最近写深度科普的方式，越来越不像"写作"了。

打开电脑，进到一个叫 `Substack-Creator/` 的文件夹。里面有十几个子目录和配置文件，没有一个是文章本身——都是规则、模板、契约、状态机协议。我启动 Claude Code，让它读 `STARTUP.md`，然后根据当前任务状态进入对应的阶段。

第一个阶段叫 planning。AI 先扮演一个叫 `AudienceAnalyst` 的角色，定义这篇文章的目标读者；然后切换成 `PoliticalTheorist`，把我模糊的想法转成可研究的政治科学问题并匹配理论框架；再切换成 `ResearchAnalyst`，做系统性深度研究，产出注释书目和带字数预算的详细大纲。每个阶段都产出一份结构化文档落在任务目录里，我逐份审阅，通过了才能推进到下一个。

然后是 writing。先 `NarrativeStrategist` 把研究大纲改造成叙事蓝图，确定每章的钩子、转场、证据锚点；再 `ContentWriter` **按章节接力写作**——一次调用只写一章，严格遵守该章的字数预算和证据要求；整篇拼接完成后由 `VoiceAndStyle` 做整体文风塑造。

再之后是 review。**三个独立视角并行工作**：`RedTeam` 找逻辑谬误和隐藏假设，`FactChecker` 做外部交叉验证并输出事实核查报告，`PeerReviewer` 以期刊审稿人立场做学术评估。三份报告产出后，一个叫 `ReviewSynthesizer` 的角色把它们聚合成结构化的决策面板，把冲突的建议显式标出来给我选择。

整个过程里，我没在"写"——至少不是传统意义上的、对着空白文档敲字的那种写。AI 也没在"写"——它只是在一套严格的契约里执行被约束好的工作。

**真正在工作的东西，是那套结构本身。**

这种工作方式有个名字，但这个名字几个月前才刚被发明。更有意思的是，这个名字当前几乎只在程序员的世界里流通。我想说的是：它其实更属于我们——属于所有做非虚构深度写作的人。

## 这门学科叫 harness engineering

"Harness"这个词在英文里有几个意思。套在马身上的缰绳和挽具，是最字面的那个。工程领域里它引申为"测试用的固定装置"——"test harness"是跑自动化测试时把被测对象固定住的一整套脚手架。

2026 年初，HashiCorp 的创始人 Mitchell Hashimoto 在一篇博客里提到一个习惯：每次 AI agent 犯错，他就在 agent 的运行环境里修一个永久性的结构性补丁，而不是去调教这一次的具体输出。他把这个做法叫做 "engineering the harness"——工程化那个套在 AI 外面的挽具。

几周之内，OpenAI 和 Anthropic 相继发文扩展这个概念。Martin Fowler 在 4 月的长文中做了学科层面的定性：**harness engineering 是"围绕 AI 模型的一切，除了模型本身"**——上下文管理、工具编排、沙箱执行、持久化状态、权限范围、错误恢复、可观测性。一个可以被写出来的公式：

> **Agent = Model + Harness**

LangChain 的工程师进一步给出了五个原语：文件系统（作为持久状态和协作界面）、代码执行（让 agent 自主解决问题）、沙箱（隔离和验证）、记忆（跨会话持久化）、上下文管理（对抗"上下文腐烂"）。

这套框架被迅速接受，因为它命名了一个大家都在做但缺乏共同语言的事。之前我们有两个相邻的术语：**prompt engineering**（优化单次对话的输入）和 **context engineering**（管理模型当前能看到什么）。Harness engineering 处理的是这两个都不负责的那一层——**当 agent 要连续工作几个小时、做几百个决策时，是什么在支撑它不偏航？**

答案不是更聪明的模型，也不是更精巧的 prompt。答案是它外面那套可审计、可版本化、可强制执行的工作环境。

## 一个盲区：所有讨论都在说代码

现在你去搜 harness engineering，能找到的所有案例、教程、最佳实践，几乎 100% 是关于 coding agent 的。

OpenAI 那篇里程碑文章描述了一个实验：三人团队、五个月、零手写代码，所有东西都由 Codex 生成。他们花大量篇幅讨论如何设计仓库结构让 agent 能"读懂"业务领域，如何把 Slack 里的架构讨论沉淀成 agent 可见的仓库文件，如何用"黄金原则"——机械化的、不可协商的规则——对抗 agent 复制不良模式造成的代码库腐烂。

Martin Fowler 的文章把 harness 分成两类组件：**guides**（前馈控制，在 agent 行动前引导它）和 **sensors**（反馈控制，在 agent 行动后观察并触发自我修正）。他举的例子是 linter、类型检查器、单元测试、CI。

Anthropic 的工程博客、LangChain 的原语分解、Vercel 的"削减工具让 agent 更可靠"案例、Meta 的多天 ML 流水线自动化——每一个具体案例都是代码、代码、代码。

这不是偶然。Coding 是第一个被 AI 规模化攻克的领域，所以它也第一个撞上"agent 能连续工作吗"这个问题，也第一个需要 harness engineering。在产业逻辑上，这很合理。

但作为一个写作者，我想指出一件反直觉的事：**写作场景——尤其是深度非虚构写作——其实比 coding 更需要 harness。而且目前几乎没有人在讨论这件事。**

## 为什么写作更需要 harness

这个判断听起来像是"写作者的自我中心"，但它有结构性的理由。

**第一，代码有天然的 sensors，写作没有。**

Fowler 的 sensor 概念——那些能在 agent 行动之后自动校验输出质量的反馈机制——在 coding 领域几乎是免费的。编译器会告诉你语法错了；单元测试会告诉你行为变了；linter 会告诉你风格偏了；类型检查器会告诉你接口不兼容了。这些 sensor 不需要你特意建造，它们是几十年软件工程积累下来的基础设施。一个哪怕设计得很粗糙的 coding harness，光靠接上这些现成的 sensor，就能获得相当可观的质量反馈回路。

写作没有这些东西。没有编译器会告诉你这段话"逻辑跳跃了"，没有单元测试会告诉你这个比喻"没击中"，没有 linter 会告诉你这个段落"正确但平庸"。中文可读性指标？Flesch 那套英文的对中文基本没意义，而且就算有，可读性和"好文章"远不是一回事。

这意味着：coding harness 可以靠复用既有 sensor 跑得挺远；**写作 harness 必须从零设计质量反馈机制**，否则就会退化成"AI 写完你看一眼"的粗糙流水线。

**第二，写作的质量判断更依赖契约，因为它不能依赖运行时验证。**

代码不确定写得好不好？跑一下。测试过了就大概率没事，性能不够就 profile 一下。写作不能"跑一下"。一篇文章"好不好"的判断，发生在读者脑子里——那是 harness 无法企及的黑箱。

这逼着写作 harness 必须在**事前**把质量要求转化成可检查的契约，而不是**事后**靠运行验证。这种"契约先行"的设计，在 coding 里也有（类型系统、接口定义），但在写作里它是**唯一的质量保障机制**，所以必须做得更严密、更显性。

**第三，写作的选题本身就可能是个错误，代码几乎不会。**

没有人会花三天写一段"根本不值得写的代码"——你要写的代码通常由业务需求驱动，值不值得写在任务启动前就已经决定了。但写作不同：**最大的浪费不是写得不好，而是选题本身不值得写**。信息密度不够、已有大量优质存量内容、无法在目标篇幅内讲清楚、或者你的视角其实并不独特——任何一条都可能让一篇认真写出来的文章从一开始就注定平庸。

这意味着写作 harness 必须包含一个 coding harness 通常不需要的东西：**选题质量门**。在 planning 之后、writing 之前，必须有显式的 go/no-go 检查点，让选题和框架在被写之前就立住。

## 我的写作 harness 长什么样

说了这么多抽象原则，具体看一眼结构更直观。这是我为深度政治学科普 Substack 搭的 harness 目录：

```
Substack-Creator/
├── CLAUDE.md / GEMINI.md      # 协调器核心指令（两份内容一致，换 agent 时无感）
├── STARTUP.md                  # 任务启动、恢复、阶段调度的状态机协议
├── agents/
│   ├── planning.md            # Planning 阶段的三个子代理
│   ├── writing.md             # Writing 阶段的三个子代理
│   ├── review.md              # Review 阶段的五个子代理（含并行设计）
│   ├── publishing.md          # Publishing 阶段的四个子代理
│   └── reflection.md          # 复盘与记忆提议
├── knowledge/
│   ├── templates/             # 契约与模板（不是知识，是 schema）
│   │   ├── subagent_call_contract.md      # 子代理调用包模板
│   │   ├── subagent_response_template.md  # 子代理返回模板
│   │   ├── workflow_state_template.md     # 工作流状态模板
│   │   ├── revision_decision_template.md  # 修订决策模板
│   │   ├── evidence_map_template.md       # 证据地图模板
│   │   └── annotated_bibliography_template.md
│   └── memory/                # 跨任务全局记忆（由 ReflectionAgent 维护）
│       ├── style_guidelines.md
│       ├── verified_sources.md
│       └── topic_map.md
├── run/                        # 每篇文章的工作目录（运行时动态生成）
│   └── 2026-04-20-ep01-.../
│       ├── workflow_state.md   # 当前任务的状态机实例
│       ├── packets/            # 每一步调用 agent 的结构化输入
│       ├── results/            # 每一步的结构化返回
│       ├── sections/           # 分章节写作产物
│       ├── draft_v1.md ... final_manuscript.md  # 各版本手稿
│       ├── *_report.md         # 各审查阶段的独立报告
│       ├── revision_decision.md
│       └── memory_update_proposal.md
└── scripts/                    # 字数校验、状态校验等机械脚本
```

这个结构背后有几个值得展开的设计决定。

### 协调器和子代理的两层分工

`CLAUDE.md`（或 `GEMINI.md`——换 AI 工具时读不同的文件，内容一致）里写着：

> 你在这个工作区里是**流程协调器**。负责与用户对话、调度阶段子代理、检查工件是否齐备、在人工审核点汇总关键内容并记录决策。

然后有一条关键的非协商规则：

> **主动提问权**：只有你可以向用户提问。子代理（或你代行时）只能返回文件、摘要、待确认问题、假设和阻塞原因。

这是整个 harness 的核心分工。协调器是和我对话的那一方，负责编排和决策汇总；子代理是被调度的执行者，只能产出结构化产物、标记阻塞、列出待确认项——**子代理不能越过协调器直接和我说话**。

这个设计看起来小，但非常重要。它避免了多 agent 系统最容易犯的错误：agent 各自和用户对话，信息分散、状态不一致、人机交互疲劳。协调器作为**唯一的对话界面**，把所有子代理的阻塞和问题聚合成一次人工决策点。

### 每一次调用都是一个结构化契约

子代理被调度时，不是收到一段自由文字的指令，而是一个完整的**调用包**（subagent call packet）。契约模板（`knowledge/templates/subagent_call_contract.md`）规定了必填字段：

```text
阶段: writing
agent: ContentWriter
执行方式: 子代理委派
任务目录: runs/2026-03-13-example-topic
输入文件:
- argument_structure.md
- detailed_outline.md
- annotated_bibliography.md
输出文件:
- sections/03-case-study.md
目标总字数:
- 7000-10000 汉字
当前估计字数:
- 4200 汉字（截至 2026-03-13 拼接稿）
章节预算:
- 01 引言：700-900 汉字
- 02 概念框架：1000-1200 汉字
- 03 核心案例：1000-1400 汉字
- 04 反方观点与回应：900-1200 汉字
- 05 结论：700-900 汉字
本轮目标:
- 完成第 03 章初稿，聚焦核心案例与机制解释
长度要求:
- 本章目标 1000-1400 汉字
- 不得以空话凑字；若证据不足以支撑下限，明确写出缺口
边界:
- 只写本章
- 不得覆写 draft_v1.md
- 不得向用户提问
证据规则:
- 保留来源键
- 证据不足时使用 [待核实]
```

这不是"建议性规则"，是 schema 级别的强制。`目标总字数`、`章节预算`、`长度要求`、`边界`、`证据规则` 全都是必填字段，缺了就阻塞。

返回也是结构化的：

```text
状态: 成功 | 阻塞 | 失败
已生成文件:
- <相对任务目录的路径>
摘要: <100-200 字>
待确认问题: - ...
默认假设: - ...
建议回滚步骤: - 无 / 步骤 N + 原因
风险或阻塞: - ...
```

这种契约化的好处是：**agent 的工作结果是可机械校验的**——`已生成文件` 列出的路径必须真实存在，`状态: 阻塞` 必须配 `风险或阻塞` 字段说明原因。协调器在推进下一步之前，可以对返回做结构校验，不符合契约就打回重做。

### 三份审查报告并行，然后由 Synthesizer 聚合

Review 阶段是整个 harness 里我最自豪的设计。传统做法是让一个 reviewer agent 承担所有审查维度（事实、逻辑、可读性）——但这会让每个维度都做得平庸，因为这些视角需要的思维模式是互相冲突的。事实核查要怀疑主义和执拗的证据追溯，红队要敌意和攻击性，学术审稿要宏观视野和结构诊断。

我的做法是让三个独立的视角**并行工作**：

- `RedTeam`：以敌对立场攻击稿件，找逻辑谬误、隐藏假设、证据不足、最容易被断章取义的表述。只输出报告，不动正文。
- `FactChecker`：对每一处具体数据、人名、日期、事件做外部交叉验证。产出四栏结构（已核实 / 已修正 / 仍有争议 / 删除弱化建议）的报告，以及只吸收"明确可证伪或可证实"修正的新版手稿。
- `PeerReviewer`：以政治学期刊审稿人立场，评估论证链、证据强度、叙事结构、章节展开深度。只输出报告。

三份报告独立产出、互不覆写。然后一个叫 `ReviewSynthesizer` 的角色把它们聚合成单一决策面板。它做三件事：

1. **去重**：指向同一问题的不同报告被合并为一个 Action Item（比如 `RT-02 + FC-W01 → ITEM-M01`）。
2. **冲突显式化**：不同报告对同一问题给出不同建议时，显式标记为冲突而不是偷偷选一个。
3. **分支化**：每个 Action Item 给出多个选项（A/B/C），标注预期影响（字数、风险、依据），推荐一个默认选项但不替用户决策。

最终产出是 `review_action_items.md`，按"已吸收 / 高优 / 中优 / 低优 / 冲突"分类，每项都有章节定位、合并来源、预计影响、默认推荐。我只需要在这一个文件上做选择，就能转化成 `revision_decision.md` 让后续 Integrator 落地。

这个架构解决了一个非常实际的问题——**多 agent 审查最大的认知负担不是读报告，是消化三份报告之间的冲突和重叠**。把这个消化动作从我这里转移到 Synthesizer 那里，我的决策时间缩短了几倍。

### 字数预算作为贯穿全流程的硬约束

如果只能挑一条 design 讲，我会挑这个——**字数预算**在我的 harness 里不是一个"写完看一眼"的指标，而是贯穿整条流水线的硬约束。

流程是这样的：

- Planning 阶段，`ResearchAnalyst` 产出 `detailed_outline.md` 时**就要锁定每章字数预算**。总预算（比如 10500-13500 汉字）拆成章节预算（01 开篇 700-1000、02 重新概念化 1200-1500、03 主框架 1500-1900......）。
- Writing 阶段，`ContentWriter` 每次只写一章，预算是硬约束。GEMINI.md 明确写着："不得靠同义反复、空泛抒情或总结性废话硬凑字数；若证据或大纲不足以支撑章节下限，返回'阻塞'并指出缺口。"
- Review 阶段，reviewer 的建议必须标注**预期字数影响**。这样 Synthesizer 聚合时能识别"这个建议会顶破 max"。
- Revision decision 阶段，每项采纳都要算预算冲突。我的一份真实 `revision_decision.md` 里是这样写的：

> ITEM-M01:A · Ch05 · 工党投票反叛=0 的解读强度限定
> 预算影响：+15~25 units，Ch05 从 1,793 → ~1,815，**会突破 max 1,800**。Integrator 必须同步从本段前半或 Meacher 段落中微量压缩，保证 Ch05 ≤ 1,800。

- Integrator 阶段落地修订时，以上述指示为精确约束执行。
- 最终产出由一个 `check_word_budget.py` 脚本做机械验证：整篇和每章是否都在 min-max 区间内。

为什么字数预算这么重要？因为**它是对"证据密度"和"论证饱和度"的代理度量**。每章 1500-1900 字的硬边界意味着 agent 不能靠空话凑字，也不能靠删减偷工。如果某章证据不足以支撑其预算，必须在 `detailed_outline.md` 的"证据缺口"字段里显式登记——**不得把问题留给写作阶段临时发挥**。这一条约束，等于在选题阶段就为每一章做了质量承诺。

在一个写作 harness 里，字数预算扮演的角色，类似于 coding harness 里的"测试覆盖率"或"性能预算"——它不是直接度量质量，而是**约束出质量的容器**。

### 记忆系统：让 harness 自我进化

最后一个我想展示的是 `ReflectionAgent` 和 `knowledge/memory/` 的设计。

每期文章发布后，`ReflectionAgent` 会读完整份 `revision_decision.md` 和所有审查报告，提炼两类东西：

- **文风准则**：从修订决策里抽取"具有全局指导意义"的规则（忽略只针对本次的局部修改）。比如我的 ep01 跑完后，它提议了一条："当正文把一项经验观察（尤其是单届/单样本/零样本）作为某个机制假说的证据时，必须附加一句不超过 20 字的限定语，显式承认'相容而非证明'。"
- **核实文献**：从事实核查报告里抽取本期新引入且被外部交叉验证的高价值文献，带标准引用、核心论点、复用价值说明。

这些提议写进 `memory_update_proposal.md`，等我批示后才追加到 `knowledge/memory/` 下的全局文件里。下一期 Planning 阶段启动时，这些全局文件会作为"专栏潜意识"被自动加载。

这个设计的关键在于：**ReflectionAgent 不直接修改记忆文件，只产出提议**。人在回路最后一道关。这样记忆库演化的速度取决于我的批示节奏，避免了"AI 擅自把不合适的规则沉淀下来"的污染风险。

更让我自己意外的是，`ReflectionAgent` 还会额外产出第四节——**关于 harness 本身的运行观察**。比如它会报告："`check_word_budget.py` 的 ASCII 词条算法会把 `ProkschSlapin2015` 等来源键合为 1 词，导致 [来源键] 剥离后总字数下降幅度小于直觉预期。建议在 CopyEditor 契约里显式说明。"

这部分不进记忆库，但进我自己的 harness changelog。它把 harness 从"静态工具"变成**自我反思、持续改进的系统**——系统不仅产出成品，还在观察自己运行中的低效点并反馈给维护者。

## 真正的设计哲学：人 in the loop，不是人 on the loop

这是我想特别讲清楚的一层，也是我认为写作 harness 最应该区别于主流 coding harness 的地方。

OpenAI 那个"五个月零手写代码"的实验展示的，是一种**人 on the loop** 的理想——人类退到外围，通过设计和维护 harness 来间接引导 agent 工作，具体的执行完全由 agent 完成。这套理想在 coding 领域是合理的：代码的质量有客观标准（能跑、正确、高性能），人类的具体介入不一定比精心设计的自动化流程更有价值。

但如果你把这套理想照搬到深度写作上，会得到一个在流程上无懈可击、在内容上毫无灵气的产物。

原因很简单：**深度非虚构写作的核心竞争力是独特视角和对复杂性的驾驭**。这两样东西没法被流程保证——流程只能保证下限，保证不了上限。如果你把所有判断都交给 agent，产出就会趋向于"规范但平庸"，因为 agent 的默认品味就是所有训练数据的平均值，而平均值恰好是最没有个性的地方。

所以写作 harness 必须坚持的是**人 in the loop**：人不是在外面监督，而是在里面做关键判断。Harness 的正确使命不是"让人越来越不用参与"，而是"**让人只在最有价值的地方参与**"。

具体到我的这套系统，人必须介入的节点是：

**选题决策点（planning 阶段的三次 gate）。** AudienceAnalyst 产出读者画像后我看一遍，PoliticalTheorist 产出理论框架后我看一遍，ResearchAnalyst 产出大纲和字数预算后我再看一遍。这三个 gate 里任何一个不过，后面全是白做——选题错了，写得再漂亮也只是一篇错文章。

**叙事骨架点（writing 早期）。** NarrativeStrategist 产出 `argument_structure.md` 后我审阅——大纲不是章节列表，是"想让读者经历什么认知旅程"。AI 擅长填充、扩写、查证，但不擅长判断"哪个比喻真的击中人"、"哪里该留白"。我在骨架上画两笔，后面 writing 的质量天花板就完全不同了。

**审稿的价值判断（review 阶段）。** 事实核查可以机械化（FactChecker + 外部搜索），但"这段虽然准确但很平庸"、"这里逻辑对但读者会走神"这种判断必须由我做。三份独立报告 + Synthesizer 聚合之后，我拿到的是一个分层的决策面板，我只做"采纳 / 拒绝 / 折中"的选择题。

**记忆沉淀的批示（reflection 阶段）。** ReflectionAgent 提议新增 5 条文风准则，我决定采纳哪几条进全局记忆。这一步决定了我的"专栏潜意识"如何演化。

对应地，agent 的舒适区是：草稿、查证、格式、一致性检查、字数控制、重复段落检测、引用规范、结构化报告产出、跨报告去重——这些是它真正擅长的事。让它做好这些，我就能把注意力集中在品味和判断上，而不是被琐碎耗竭。

一个具体的 harness 技巧：在状态机里，显式标注哪些状态转换是 `human_gate: true`。比如 `planning_done → writing_start` 必须人工确认，agent 看到 gate 就停下来等我输入，不能擅自推进。这个小小的结构决定，把"人应该在哪里介入"从自觉变成了强制，避免了偷懒（我自己会偷懒）和漂移（agent 自己会漂移）。

## 与 prompt 和 skill 的区别

写到这里，熟悉 AI 工具的读者可能会问：**这和 prompt、和 skill、和自定义 GPT 有什么本质区别？**

这是好问题，值得专门澄清。

**Prompt engineering** 优化的是单次对话的输入——给模型什么指令能得到更好的输出。它是 harness 的一个组件，但不等于 harness。Fowler 的比喻很准：如果 prompt engineering 是"告诉司机向右转"，harness engineering 是"道路、护栏、交通标志、信号灯共同构成的让十辆车安全导航的系统"。

**Skill**（比如 Claude 的 skill 机制）封装的是**能力**——一个特定任务 agent 可以加载的便携工具包。它适合"这件事偶尔做一下、需要时加载"的场景：格式转换、特定 API 调用规范、领域术语处理。skill 是 agent 工具箱里的零件，由 agent 自己判断何时使用。

**Harness** 封装的是**工作方式**。它不是某个能力，而是一整套带状态、带流程、带人工节点、带产物沉淀的工作环境。主动权不在 agent，而在你——你决定进哪个文件夹、跑哪个阶段、什么时候介入。Skill 问的是"agent 需要什么能力"，harness 问的是"这件事应该怎么被做"。

我的 `CLAUDE.md` 里有一条来之不易的警告：

> 本工作区已完全包含了专栏创作的完整流程。在此工作区内，严禁调用 `activate_skill` 工具去激活全局的 `substack` 技能，以免造成规则重复和上下文污染。

这是我踩坑之后加的。Skill 和 harness 如果同时作用，规则会在 agent 的上下文里重叠、冲突、互相覆盖，产出质量反而下降。两层抽象必须**互相独立**，harness 管主干流程和人工节点，skill 管具体的原子能力，不要混用。

但对写作者来说，如果你要选一个起点，选 harness 而不是 skill。因为**你真正需要沉淀的是"深度写作这件事应该怎么被做"——这是一套工作方式，不是一个或一组技能**。Skill 装不下它，只有 harness 能。

## 过度工程化的陷阱

我必须诚实地讲这部分，因为它是 harness 最大的风险。

Harness 这套结构一旦搭起来，会产生一种**结构带来的愉悦感**——你会享受流程的整齐、模板的精密、脚本的自动化。这种愉悦本身没问题，问题是它会掩盖两件事：

**第一，harness 的维护成本可能超过它的收益。**

这套系统适合的是"需要稳定产出、有一定批量"的场景——如果你一年要写 50 篇深度文章，harness 几乎一定是值得的。但如果你一个月才写一两篇，每次写作前对着 harness 调一调提示词、改一改模板、修一修脚本的时间，可能已经超过你直接手写那一两篇的时间。

一个自检问题：**上次你改 harness 花的时间，如果直接用来手写文章，是不是产出更高？** 如果答案连续几次都是"是"，说明你在享受建造工具而不是享受产出内容。这是一个陷阱。

**第二，harness 容易变成"安全毯"。**

流程越规整，越容易产生"按流程走完 = 作品合格"的错觉。你会开始迁就流程——因为某个 agent 不擅长处理某类选题，就避开那类选题；因为某个模板不支持某种叙事结构，就不用那种结构。这时候 harness 不是在服务你，是在塑造你。

判断这件事的信号很微妙：如果你发现自己产出的文章都"挺规范但没什么记忆点"——没有那种让人看完想转发、想引用、想反驳的东西——那就是 harness 在压制而不是放大你的判断力。这时候应该把 harness 用得更轻，只保留事实校验、证据政策、字数控制这些**机械性**的部分，把选题视角、叙事结构、语言节奏这些**创造性**的部分交还给人。

Harness 的正确使用姿势不是"越完整越好"，而是**识别出哪些环节适合流程化、哪些必须保留人的判断**。这个识别是动态的——随着你对写作的理解变化、随着模型能力变化、随着你自己状态变化，边界会移动。所以 harness 本身也需要被定期审视和重构。

## 为什么现在应该开始

最后说一下为什么我认为写作者现在就该开始做自己的 harness，而不是再等两年看看。

**第一，harness 独立于 agent。** 这是它最被低估的价值。今天你用 Claude Code、明天换别的——只要新 agent 能读 markdown 指令、能访问文件系统、能执行脚本，你的 harness 就能继续用。我的 `CLAUDE.md` 和 `GEMINI.md` 内容完全一致就是这个设计的直接体现——同一套规则，两份镜像文件，换 AI 工具时零迁移成本。你沉淀下来的不是对某个 AI 工具的熟练度，而是**你自己的写作方法论的可执行版本**。这是真正能穿越模型换代周期的资产。

**第二，模型会越来越强，但你的品味是稀缺品。** 很多人担心"把流程做得这么精细，将来模型能力上去了不就白做了？"这个担心其实搞反了方向。模型会越来越强这件事，恰好意味着**能跟它协作的人越来越多**，同质化也会越来越严重——每个人都能用 AI 产出"还不错"的内容，意味着"还不错"已经不构成任何竞争力。真正稀缺的是**你独特的判断力**——什么选题值得做、什么角度有张力、什么段落该删、什么比喻能击中人。Harness 是把这种判断力固化、复用、放大的载体。模型越强，harness 放大出来的判断力价值越高。

**第三，写下来的流程会反过来改进流程本身。** Harness 最让我意外的副产品是：**它逼我把默会的写作直觉变成显性的工作准则**。"什么是一个好选题"这个问题，我以前只能靠手感回答；被迫写进 `planning.md` 和 `GEMINI.md` 之后，我自己对这个问题的理解也更清晰了。更有意思的是，ReflectionAgent 每期提议的文风准则，会从我的修订决策里抽取出**我自己都没意识到的模式**——比如"限定句应为独立小句而非插入式"这条规则，是我一直在做但从没说出口的偏好。harness 把这种默会知识捞了上来。

## 结尾：一门属于创作者的新手艺

Harness engineering 当前被定义为一门工程学科，主流讨论也确实在工程语境里发生。但学科的出发点——**让 AI 在一套精心设计的约束中做有价值的工作，让人只在最需要判断的地方出现**——这个哲学本身和工程无关，和创作的关系反而更深。

写作者一直在做类似的事：我们用写作指南约束自己的语言，用风格手册保持出版物的一致性，用审稿流程拦住质量下限，用编辑对话发现自己看不见的盲点。这些都是 harness，只是以前它们是人和人之间的协作结构。现在有了 AI，这些结构的一部分可以被**形式化、版本化、自动化**，并且和一个不会疲倦的执行者对接。

但 harness 的灵魂不是自动化。是**你作为创作者的方法论变得可被审视、可被迭代、可被持续改进**。这件事在 AI 出现之前就有价值，现在只是有了一个更强的载体。

所以如果你是一个认真对待写作的人，不要把 harness engineering 当成程序员的小玩具。它是一门**属于所有需要在复杂性中保持判断力的创作者**的新手艺。

从哪里开始？从你下一篇文章开始。别搭一个宏大的结构——搭最小的那个：一个 `writing.md` 写清楚你的写作原则，一个 `checklist.md` 列出你审稿时检查的维度，一个 `runs/` 目录存中间稿。就这三样，你已经有一个 harness 了。

剩下的，随着你写的每一篇文章长出来。
